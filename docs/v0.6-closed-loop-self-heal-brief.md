# Implementation Brief: PandaProbe Harness — Closing the Self-Heal Loop (v0.6)

## 0. Read this first

You are extending an existing, working Python framework. **Do not rewrite it.** Read the
current code before you change it, reuse its patterns, and keep its invariants. The whole point
of this work is to turn an *open-loop* self-healing system (it detects failures, proposes rules,
and applies them, but never confirms the rules actually help) into a *closed-loop* one (it
validates that a learned rule improves behavior before trusting it, retrieves only the rules
relevant to the current situation, and continuously guards against regressions). All of this must
happen **without a human in the loop** — the agent and the harness do the healing and the
validation themselves.

## 1. What this framework is (context)

**PandaProbe Harness** turns any LLM agent into a **self-healing** agent. It is the operational
envelope around a model — a diagnostic workspace on disk, a toolbelt, standing rules, and
lifecycle hooks — that lets the agent observe its own failures, analyze them, and rewrite its own
operating rules with minimal human intervention.

It sits on top of two pre-existing tools it does **not** own:

- the **PandaProbe Core SDK** (passive tracing of agent runs), and
- the **PandaProbe CLI** (`pandaprobe`, a Go binary wrapping the evaluation/telemetry REST API).

The harness adds the *harness layer* on top. Its north star is a **workspace-pull model**: rather
than pushing alerts into the agent's next input, the harness maintains a **mailbox** and a
**journal** in a `/harness` workspace, surfaces a tiny "you have N pending notices" banner through
the always-loaded system prompt, and gives the agent a first-class **toolset** to read its
mailbox, inspect its own traces, analyze failures across runs, and rewrite its own rules — then
acknowledge the notice. The agent drives the loop; the harness maintains the substrate and the
trigger.

### The loop today (v0.5)

```
turn end ──▶ hook.on_turn_end ──▶ [budget→sample→rate-limit→supersede] ──▶ _run_eval task
  _run_eval: health-gate → optional backend history hydration → (semaphore) evaluate_turn
             → EWMA trend detection → dedup/cooldown gate → circuit breaker
             → build DiagnosticNotice (sanitized) → write traces/<id>.json + latest_eval.json
             → Mailbox.post(notice) → Journal.record("notice")
agent turn start: system context = active rules + pull protocol + "⚠ N pending notices" banner
  agent: harness_mailbox_list → harness_mailbox_read → harness_trace_inspect
         → harness_journal/harness_history → harness_rule_add(rule, rationale, notice_id)
         → harness_mailbox_ack → (periodically) harness_reflect
```

Two session-level metrics drive everything, computed by the platform per `session_id`:
`agent_reliability` (trajectory risk) and `agent_consistency` (session stability), each in
`[0.0, 1.0]`, higher is better, breach below `0.5` by default.

### The gap this brief closes

The v0.5 loop is **open-loop**: it detects → proposes → applies, but never *confirms*. Concretely:

1. A self-authored rule is trusted the moment it is written; nothing checks it actually reduces
   failures. `RulesStore.effectiveness()` is computed *after the fact* from notice counts on the
   same noisy metric that proposed the rule — circular, not evidence.
2. Every active rule is injected into *every* prompt for *every* task, so a rule learned from one
   failure mode dilutes attention and can harm unrelated tasks (learned-constraint overfitting).
3. Nothing catches a *regression*: a new rule that quietly breaks a previously-passing scenario.
4. There is no way for the operator to know whether the `breach` trigger even correlates with real
   failure — the threshold is a guess.

This brief adds four capabilities that turn the loop closed. **Item 5 from the original
assessment (a human review tier) is intentionally excluded** — validation must be automatic and
agent/harness-driven.

## 4. The hard constraint that shapes items 1 & 4 (read carefully)

The PandaProbe platform is **passive and trace-based**: it scores the traces a session already
produced. You **cannot** retroactively re-score a past session "as if a rule had been in effect" —
the traces are fixed. Therefore, the only way to get *counterfactual evidence* that a rule helps is
to **replay** the scenario: re-run the agent on the same input with the candidate rule in its
system context, producing *new* traces under a *new* session id, then score those.

The harness cannot run the user's agent itself. So replay requires a **developer-supplied replay
function**. This is the backbone of both candidate validation (item 1) and the regression eval-set
(item 4). When no replay function is wired, the harness falls back to **forward trial** (observe
the rule's effect on future live sessions) — which needs no replay and no human, satisfying the
"fully agent-driven" requirement, at the cost of slower, statistical evidence.

Design both paths. Make replay the strong path and forward-trial the automatic fallback.

## 5. The four items

### Item 1 — Candidate-rule validation before promotion (highest priority)

**Goal:** a newly authored rule is not trusted until there is evidence it improves the metric it
targets. Turn "the agent wrote a rule" into "the rule demonstrably helps."

**Rule lifecycle.** Extend `Rule.status` to the state machine:

```
candidate ──(validated: metric improved)──▶ active
    │
    └────────(invalidated: no improvement / regressed)──▶ retired
active ──(agent or regression run retires it)──▶ retired
```

- `harness_rule_add` creates a rule as **`candidate`** by default (not `active`).
- A candidate **is** injected into the system context (so it can take effect and be measured), but
  rendered under a clearly-labeled "Provisional rules (under evaluation)" heading in
  `render_markdown()`, distinct from validated `active` rules. This is important: a candidate must
  be in force to be testable, but the agent (and any reader) must know it is unproven.
- Validation is performed by a pluggable **`RuleValidator`** (new module
  `src/pandaprobe_harness/validation/`), with two built-in strategies:
  1. **`ReplayValidator`** (strong; used when a replay function + a matching eval case exist,
     see item 4): replay the failing scenario(s) whose signature matches the candidate's, with the
     candidate rule injected into the replayed system context; score the new session(s) via
     `MetricEvaluator`; **promote to `active` iff** the targeted metric improves past a margin
     (`rule_promote_margin`) **and** no other case in the sampled set regresses beyond
     `rule_regress_margin`. Otherwise **retire** the candidate with a journaled reason.
  2. **`ForwardTrialValidator`** (automatic fallback; no replay required): track the candidate over
     the next `rule_trial_min_sessions` distinct sessions that carry its metric/signature. Compare
     the breach rate for that signature during the trial against the pre-candidate baseline (the
     signature's recent breach rate from the journal, captured at add time and stored on the rule).
     Promote if the rate dropped (or is zero); retire if it did not improve after the window.
- Add a `Rule` field `trial: dict` (or dedicated fields) holding `{baseline_breaches, baseline_window,
  trial_started_at, observed_sessions, observed_breaches, verdict}` — forgiving JSON, defaulted.
- The harness drives validation automatically (no human): run candidate evaluation from a new hook
  step (e.g. after `_persist_notice`, and/or a periodic `_evaluate_candidates()` invoked on a turn
  cadence). Every promotion/retirement is a journal event (`rule_promote`, `rule_retire` with a
  `reason`), so `harness_reflect` can learn from validation outcomes.
- New toolset op **`harness_rule_status(rule_id)`** returning the rule's lifecycle state + trial
  bookkeeping, so the agent can see why a rule was promoted/retired.

**Config knobs (in `config.py` + `from_env`):** `rule_validation: bool = True`,
`rule_trial_min_sessions: int = 5`, `rule_promote_margin: float = 0.05`,
`rule_regress_margin: float = 0.05`. When `rule_validation=False`, restore v0.5 behavior (rules
enter `active` immediately) — this keeps the simple offline path trivial and is the compatibility
switch for the existing e2e test if you choose not to rewire it (but prefer rewiring — see tests).

**Files:** `config.py`, `workspace/rules.py` (status enum, provisional rendering, trial fields,
baseline capture on `add`), new `validation/__init__.py` + `validation/validator.py`
(`RuleValidator` protocol + the two strategies), `hook/core.py` (candidate-evaluation step + wiring
of the validator/replay seam), `hook/context.py` (provisional heading), `agent_tools/toolset.py`
(`harness_rule_status`, and `harness_rule_add` now returns the `candidate` state), `harness.py`
(accept an optional `replay`/validator on `create`).

### Item 4 — Replayable regression eval-set (build alongside item 1; they share the replay seam)

**Goal:** accumulate a corpus of scenarios (failures and known-good "wins") that can be re-run
against the *current* rule set, so (a) item 1 has something to replay, and (b) a new rule that
breaks an old win is caught automatically.

**New workspace store** `src/pandaprobe_harness/workspace/evalset.py` (`EvalSet`), backed by
`<harness_root>/evalset/` (per-case JSON files or one JSONL — match the mailbox/rules style). Add
derived paths to `config.py`. `EvalCase` (frozen dataclass, forgiving JSON):

```
id, created_at, session_id, kind: "failure" | "win",
signature: tuple[str, ...],          # copied from the notice that spawned it
baseline_scores: dict[str, float],   # {metric: value} at capture time
replay_input: Any | None,            # the payload a replay fn needs (developer-attached)
notes: str
```

- **Population.** When a breach notice is posted, if `capture_eval_cases` is on, capture the session
  as a `failure` case (store `session_id`, `signature`, `baseline_scores` from the report). The
  *replay input* (the user's prompt/task) is not visible to the harness in general — capture it
  from `TurnContext.end_state` when present, and expose an API/tool to attach it explicitly
  (`harness.evalset.attach_input(case_id, payload)` / a toolset op). Cap the corpus at
  `eval_case_max` (evict oldest failures first; never evict `win` cases automatically).
- **Replay seam.** Define `ReplayFn = Callable[[EvalCase, str], Awaitable[str]]` — given a case and
  a system-context string, re-run the agent and return the **new session id** it produced. The
  harness then scores that session via `MetricEvaluator.evaluate_turn` (build a synthetic
  `TurnContext` for the new session id) and compares to `baseline_scores`. Wire the optional
  `replay` on `Harness.create(..., replay=...)` and on the hook.
- **Regression run.** Add `await harness.run_regression(*, sample=None) -> RegressionReport` and an
  operator CLI `pandaprobe-harness-eval` (new `[project.scripts]` entry, or a subcommand of the
  companion CLI — your call, keep it stdlib): replay all (or a sampled subset of) eval cases
  against the **current** rendered rule set, score them, and report per-case
  `improved / unchanged / regressed` vs. baseline plus an aggregate. Journal a `regression` event.
  This is the periodic guard. Promote it in the README as the "did a new rule break an old win"
  check.
- **Integration with item 1.** `ReplayValidator` calls the same replay seam, but scoped to the
  case(s) whose `signature` matches the candidate rule, plus a small sample of `win` cases to catch
  collateral regressions.

**Config knobs:** `capture_eval_cases: bool = False` (opt-in; it stores session data),
`eval_case_max: int = 200`, `regression_sample: int = 0` (0 = all).

**Files:** `config.py`, `workspace/evalset.py`, `workspace/__init__.py` (export), `hook/core.py`
(capture on breach), `harness.py` (`run_regression`, `replay` wiring, `evalset` property),
`agent_tools/` (optional op to attach a replay input / list eval cases), new console script.

**Honesty requirement:** the replay dependency must be documented prominently. Without a `ReplayFn`,
`run_regression` and `ReplayValidator` cannot run — the harness must log this clearly once and fall
back to forward-trial validation, never silently no-op.

### Item 3 — Task-conditioned rule retrieval

**Goal:** stop dumping every active rule into every prompt. Inject only the rules relevant to the
current situation, so rules don't dilute attention or overfit across tasks. Zero-dependency,
lexical/statistical relevance only.

- **Tagging.** Add `tags: tuple[str, ...]` to `Rule`. On `harness_rule_add`, auto-derive tags from
  the source notice when `notice_id` is given (the notice's `signatures`, `metrics[].name`, and any
  tool names extractable from `flagged_traces`/`signal_breakdown`), and accept explicit `tags`.
  Untagged rules are treated as **global** (always eligible).
- **Retrieval in `compose_system_preamble`.** When `rule_retrieval` is on, render:
  (a) all global/untagged rules (always), plus (b) the top-`rules_context_topk` tagged rules ranked
  by lexical relevance to a **query**. The query is assembled from the current signal available at
  context-composition time: the pending notices' `signatures` + metric names, and an optional
  caller-supplied **task hint** (add an optional `task_hint: str | None` parameter to
  `system_context()`/`compose_system_preamble`). Relevance = a stdlib lexical score (token-overlap
  / Jaccard over rule text+tags vs. query, or a small BM25-lite); do NOT pull in an embedding
  library. Provisional (candidate) rules from item 1 still render in their own section regardless of
  retrieval (they must stay in force to be validated).
- **On-demand access.** Add toolset op **`harness_rules_search(query, limit=10)`** so the agent can
  retrieve rules beyond the top-k when it needs them, and **`harness_rules_list(status=None)`** for
  a full listing. This keeps the context small while keeping every rule reachable.
- When `rule_retrieval=False`, behave as v0.5 (render all active rules, capped by
  `max_active_rules`).

**Config knobs:** `rule_retrieval: bool = True`, `rules_context_topk: int = 8`.

**Files:** `workspace/rules.py` (`tags`, tag derivation, a `search(query, limit)` /
`relevant(query, k)` method with the lexical scorer), `hook/context.py` (retrieval-aware
rendering + `task_hint`), `harness.py` (`system_context(task_hint=None)`), `agent_tools/toolset.py`
(`harness_rules_search`, `harness_rules_list`), `config.py`.

### Item 2 — Metric calibration tool (offline diagnostic; not a runtime gate)

**Goal:** let the operator answer "does a `breach` actually correspond to a real failure, and is my
threshold sane?" This does **not** put a human in the healing loop — it is an out-of-band analysis
utility. It gives the operator the evidence to trust (or retune) the trigger that everything else
depends on.

- New module `src/pandaprobe_harness/calibration.py` + operator CLI
  `pandaprobe-harness-calibrate` (new `[project.scripts]`, stdlib only). Data comes through the
  `CliClient` seam (`evals scores list --target session`) and/or the local `ScoreHistoryStore` and
  `EvalSet` — never the REST API directly.
- **With labels** (optional): the operator supplies ground-truth labels (`session_id -> failed:
  bool`) via a JSON/CSV path, or the labels are read from `EvalSet` (`kind` = failure/win as a
  proxy label). Produce **precision / recall / F1** of `breach` (score < threshold) vs. label, a
  confusion matrix, and a **threshold sweep** recommending the threshold that maximizes F1 (and one
  that hits a target precision). Per metric.
- **Without labels** (unsupervised): produce the score **distribution** (min/median/mean/max,
  simple histogram buckets), the breach count at each candidate threshold (a sweep), and
  inter-metric agreement. Enough for the operator to pick a threshold sanely.
- Output a frozen `CalibrationReport` dataclass and a human-readable text/JSON rendering. All stats
  in stdlib (`statistics`, hand-rolled histogram; no numpy).

**Files:** `calibration.py`, a console script, tests, an example, README section.

## 8. Design decisions to make explicitly (resolve, don't hand-wave)

State your choice and a one-line rationale for each in your final report:

- Candidate rendering: how a provisional rule is visually separated in `render_markdown` and
  whether the agent is told (in `_PULL_PROTOCOL`) to treat provisional rules as tentative.
- Forward-trial baseline: exactly how the pre-candidate breach rate is captured (which journal
  window, per-signature vs. per-metric) and the promotion/retirement thresholds.
- Replay scoring: how you synthesize a `TurnContext` for a replayed session id and feed it to
  `MetricEvaluator` without disturbing the live `_pending`/refresh bookkeeping (do NOT route replay
  scoring through `on_turn_end`; call the evaluator directly).
- Retrieval scorer: the exact lexical formula and how ties/empties are handled; confirm zero deps.
- Eval-case input capture: where `replay_input` comes from and what happens when it is absent
  (the case is still storable for calibration/labels but not replayable — log/skip cleanly).
- Calibration labels: the file format(s) accepted and the `EvalSet`-as-proxy-label path.
- Concurrency: candidate evaluation and regression runs must not block the host loop or corrupt the
  stores under the existing thread-pool + lock model; state where you run them

## 9. Acceptance criteria

- `make test` (offline), `make lint` (ruff), `make typecheck` (`mypy --strict` over `src`) all
  green; core stays **zero runtime dependencies**; new deps (if any) only behind extras.
- A rule authored via `harness_rule_add` is created as `candidate`, and is promoted to `active`
  **only** after `ReplayValidator` (with a replay fn) or `ForwardTrialValidator` (without one)
  produces a favorable verdict; unfavorable candidates are retired. All transitions journaled.
- `compose_system_preamble` injects global rules + the top-k retrieved rules for a query (not the
  full set) when retrieval is on, and reproduces v0.5 rendering when off; `harness_rules_search`
  reaches the rest.
- `harness.run_regression()` (and its CLI) replays eval cases against the current rule set and

  log line, never a crash.
- `pandaprobe-harness-calibrate` produces precision/recall + a threshold sweep with labels, and a
  distribution + sweep without labels, entirely offline in tests.
- The closed-loop e2e proves: breach → candidate → validated-by-replay → promoted → retrieved into
  context → regression-clean. The cross-run journal shows `notice → rule_add → rule_promote → …`.
- README documents the replay-function requirement and the forward-trial fallback honestly; version
  is `0.6.0` in both files; `CHANGELOG.md` has a `0.6.0` entry with an explicit note that
